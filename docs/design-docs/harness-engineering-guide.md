# Harness Engineering Guide

## Scaffold Provenance

The initial scaffold, including repository structure, CI configuration, formatting rules, package manager setup, and application framework defaults, was generated by Codex CLI using GPT-5 and a small set of existing templates. The initial `AGENTS.md` file that directs agents how to work in the repository was also written by Codex.

## Monorepo Policy

This repository should remain compatible with a mixed-language monorepo by default.

Preferred layout:

- `apps/` for deployable applications
- `packages/` for shared TypeScript libraries and UI packages
- `crates/` for Rust crates and CLIs
- `python/` for Python services, tools, or libraries
- `supabase/` for local Supabase configuration and migrations when used

When a new codebase is scaffolded, prefer monorepo-friendly tooling and paths from the start instead of single-app layouts that are expensive to restructure later.

## Default Stack Profiles

### Web App

For a frontend or web application, default to:

- Next.js
- React
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase
- tRPC
- Zustand
- WorkOS

Preferred monorepo target:

- `apps/web`

Preferred package manager for TypeScript workspaces:

- `pnpm`

### Desktop App

For a desktop application, default to:

- Tauri
- The same frontend stack as the web app profile

Preferred monorepo target:

- `apps/desktop`

### Mobile App (iOS)

For an iOS mobile application, default to:

- Swift
- SwiftUI
- XcodeGen for deterministic project generation

Preferred monorepo target:

- `apps/ios`

### Rust App

For a Rust application, default to:

- `clap`
- `ciclack`-style CLI prompts when interactive UX is needed
- `tokio` only when async IO or concurrency is actually required

Preferred monorepo target:

- `crates/<app-name>`

### Python App

For a Python application, default to:

- A local `.venv`
- Type-checked Python code
- Repo-local formatting and linting commands

Preferred monorepo target:

- `python/<app-name>`

## Quality Gates

All generated stacks should include formatter and linter support from the beginning, and CI should enforce them.

Minimum expectation:

- A formatter command
- A linter command
- A type-check command where the language supports it
- A CI workflow that runs the highest-signal validation path for the scaffolded stack

For TypeScript stacks, prefer commands that cover lint, format, and typecheck. For Rust, prefer `cargo fmt`, `cargo clippy`, and `cargo test` or `cargo check` as appropriate. For Python, prefer formatter, linter, and type-check commands in addition to test or smoke validation.

## Setup Workflow Expectations

When a user asks to set up a codebase:

1. Infer the likely stack from the prompt when possible.
2. If the prompt is ambiguous, ask only the minimum stack-selection questions required.
3. Prefer repo-owned templates and `scripts/bootstrap_codebase.py` when the requested stack is supported.
4. Choose a monorepo-friendly scaffold path.
5. Add linting, formatting, and CI expectations to the setup plan.
6. Avoid inventing stacks outside the defaults unless the user explicitly requests an exception.

## Deterministic Bootstrap

The repository should move setup logic from recommendations into versioned templates and scripts.

Current bootstrap entrypoint:

- `python3 scripts/bootstrap_codebase.py --stack <stack> --name <name> --target <path>`

Current repo-owned supported stacks:

- `ios-swiftui`
- `web-nextjs`

Bootstrap output should be deterministic, monorepo-friendly, and include formatter, linter, typecheck, and CI coverage from the first generated commit.
